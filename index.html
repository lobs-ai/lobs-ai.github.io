The Origin

One dev.
An entire team of agents.

What started as a simple OpenClaw assistant evolved into a full multi-agent orchestration platform — driven by real pain points at every stage.

Built by Rafe — a UMich CSE grad student who juggles classes, GSI duties, and a summer Microsoft internship — all while operating like a one-person startup. The goal was never just an AI assistant. It was a workforce — specialized agents that run autonomously, pick up tasks, and deliver results while he's in lecture or asleep.

Three iterations. Each one forced by a hard ceiling hit in production.

v1

Simple OpenClaw Assistant

Single agent, manual prompts, basic tool use. The foundation.

⚠️ Why it wasn't enough

Hit ceiling fast. Single agent couldn't handle complex multi-step work. No memory persistence — every session started cold. No task management meant nothing could run while Rafe was away. The moment you needed a chain of actions, you were stuck.

↓
v2

lobs-server + Python Orchestrator

FastAPI backend with a Python-based multi-agent orchestrator. Actually worked — for a while.

⚠️ Why it broke down

38,000 lines of Python meant bugs were genuinely hard to track. aiohttp session leaks under load. Spawn race conditions where two workers would grab the same task. No proper workflow engine — just a pile of if/else chains. The glue was heavier than the logic it connected.

↓
v3

PAW Plugin — TypeScript Rewrite

OpenClaw's plugin API was the insight. Why build a separate server when you can extend the runtime directly?

✅ The breakthrough

79% code reduction — 38K lines → 8K lines — because most of the "glue" disappeared. The plugin runs inside OpenClaw. No separate process, no IPC, no session handshake overhead. Workflow engine, model chooser, circuit breaker — all first-class. The system became more capable, not less.

38,000 → 8,000 lines 79% smaller

End-to-End Flow

How It Actually Works

From a message to a completed deliverable — six steps, zero manual intervention after the first.

01

Rafe sends a message or creates a task

Via Discord DM, iOS app, or directly in the task database. Natural language or structured input — both work.

02

Lobs (PM agent) triages and routes

Reads the request, determines urgency and agent type, breaks it into subtasks if needed, and writes it to the PAW task queue with the right model tier.

03

PAW orchestrator matches to a workflow

The control loop (runs every 10s) picks up active tasks, matches them to workflow definitions if applicable, and sequences the work across multiple agent types.

04

Model chooser selects the right tier

Based on task complexity, agent type, and cost budget. Circuit breaker kicks in if a model fails — automatic fallback up the tier chain.

05

Worker agent spawns and executes

A sandboxed OpenClaw session fires up with the appropriate agent config, workspace, and tool access. Max 2 concurrent workers. 15-minute timeout enforced.

06

Results captured, task completed, learning stored

Worker output lands in the task DB. Lessons learned get fed into shared memory. Rafe gets a quiet notification — only if it matters.

bash — create a task directly 
# Insert a task into the PAW DB — orchestrator picks it up within 10s
sqlite3 ~/.openclaw/plugins/paw/paw.db \
  "INSERT INTO tasks
     (id, title, status, agent, model_tier, notes, created_at, updated_at)
   VALUES (
     lower(hex(randomblob(16))),
     'Refactor auth module to use JWT',
     'active', 'programmer', 'standard',
     'Follow existing patterns in /src/auth. Add tests.',
     datetime('now'), datetime('now')
   );"

# Worker spawns, codes, commits, reports back. Zero babysitting.

Build Log

Wins & Hard-Won Lessons

Real systems break in real ways. Every incident below happened in production. None were fun. All were instructive.

🏆 Wins & Milestones

⚙️

Workflow Engine as Control Plane

Multi-step tasks now run as structured workflows with state, branching, and rollback — not one-shot prompts. Enabled work that previously required constant babysitting to run fully unattended.

📉

79% Code Reduction

Python → TypeScript rewrite. 38K lines became 8K. The system became more capable, not less. Every deleted line was infrastructure glue that the plugin architecture made unnecessary.

🎯

Model Tier System

Tasks automatically route to the cheapest model that can handle them. First week of operation: 60% of tasks handled by the free local model.

🪞

Agent Reflection & Learning

Workers reflect on completed tasks, capture lessons, and feed back into shared memory. The system gets meaningfully better week-over-week without manual updates.

🔌

Circuit Breaker Patterns

Automatic failure detection prevents cascading issues. Workers that fail too often get quarantined. Saved the system from multiple potential death spirals in the v2 era.

💀 Losses & War Stories

🔄

The Restart Loop Incident HIGH

Workers edited plugin source code, then called gateway restart. The restart spawned fresh workers. Who picked up active tasks. Who edited source and called restart. Infinite loop. The system was restarting itself every 30 seconds for 20 minutes before it was caught.

💡 What we learned

Gateway restart is now hard-denied in all worker agent tool configs. Workers can read source but cannot modify it. Defense in depth over clever permission systems.

💬

Heartbeat Spam MEDIUM

Background exec commands in workers triggered event-driven heartbeats on completion — flooding the main session with "Exec completed" noise. At peak, 40+ heartbeat messages in 10 minutes. Completely drowned out actual notifications.

💡 What we learned

Banned all background exec patterns (&, nohup, sleep &&) in worker configs. Direct, synchronous commands only. The AGENTS.md now has a whole section on this.

🐛

JSON Double-Conversion Bug MEDIUM

Swift's .convertFromSnakeCase was set on the decoder. Then manual CodingKeys were added for the same snake_case → camelCase mapping. The decoder applied both. Fields silently dropped. No errors — just missing data. Took two sessions to diagnose.

💡 What we learned

Never add CodingKeys for simple snake→camel conversions when .convertFromSnakeCase is active. Now documented as permanent institutional knowledge in TOOLS.md.

👻

Orphaned Session Errors LOW

"No tool call found" errors from workers that completed their task without making any tool calls. Session validation expected at least one tool call — workers that answered purely from context would silently fail and get retried, wasting tokens.

💡 What we learned

Stricter session validation distinguishing "no tools needed" from "something went wrong." Simple tasks that don't need tools are now explicitly tagged at creation time.

🔁

Reflection Spam LOW

Agent reflection was too eager — workers would reflect after every minor action: reading a file, running a test, making a small edit. Shared memory grew by hundreds of low-quality entries in a single day. Signal buried in noise.

💡 What we learned

Reflection now gates behind strict completion criteria and rate limits. Only significant task completions trigger reflection. Quality over volume, every time.
